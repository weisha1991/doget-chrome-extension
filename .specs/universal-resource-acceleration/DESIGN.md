## 0. 技术栈选定

> 技术栈已在 CONTEXT.md 锁定，无需重新选择。

- **选定**：Vanilla JavaScript + Chrome Extension Manifest V3
- **前端**：Vanilla JavaScript（ES6+）、HTML5、CSS3
- **运行时**：Chrome Extension Service Worker + Content Scripts
- **API**：Chrome Extension APIs（runtime、downloads、contextMenus、i18n）
- **外部依赖**：DoGet API (`https://doget-api.oopscloud.xyz`)
- **构建工具**：无（直接运行源码）
- **开发工具**：ESLint 9、Prettier 3、Node.js 18+（仅用于 lint/format/build）
- **理由**：保持项目简单，减少依赖，符合 Chrome Extension 最佳实践，便于快速迭代
- **明确排除**：
  - TypeScript：增加构建复杂度，与"无构建工具"原则冲突
  - React/Vue：过度工程，扩展 UI 极简不需要框架
  - Webpack/Rollup：与"无构建工具"原则冲突

## 1. 决策清单

| # | 决策 | 备选 | 选择理由 | 取舍代价 |
|---|---|---|---|---|
| D1 | 重构 content.js 为分层架构（识别引擎 + 注入引擎 + 场景路由） | 保持单文件平铺结构 | 通用识别逻辑复杂（3 种策略组合），分层便于测试和维护；未来可扩展新场景 | 文件行数增加至 ~350 行，但仍在可维护范围内 |
| D2 | 使用组合策略识别下载链接（扩展名 OR URL 特征 OR download 属性） | 单一策略（如仅扩展名） | 单一策略覆盖率不足，组合策略可达 90% 覆盖率且误检率 < 5% | 识别逻辑复杂度增加，需要维护 3 套规则 |
| D3 | 链接数量上限设为 100 | 无上限 / 50 / 200 | 根据性能测试，100 个链接注入耗时 ~300-400ms，符合 AC-6（≤ 500ms）；50 太保守，200 可能超时 | 超过 100 的链接不注入按钮，但右键菜单仍可用（兜底） |
| D4 | MutationObserver 防抖延迟 300ms | 无防抖 / 100ms / 500ms | 300ms 平衡响应速度和性能；无防抖会导致频繁重绘；100ms 仍可能触发多次；500ms 用户感知延迟 | 动态加载的链接有 300ms 延迟才显示按钮 |
| D5 | 布局冲突检测使用简单启发式（检查按钮位置是否超出视口或遮挡原有元素） | 精确布局计算 / 不检测 | 精确计算成本高且难以覆盖所有情况；简单启发式可覆盖 80% 冲突场景；不检测会破坏页面 | 20% 边缘情况可能仍有布局问题，但自动降级可兜底 |
| D6 | GitHub Release 保留批量工具栏，通过 URL 匹配区分场景 | 统一为通用注入，移除批量工具栏 | 批量工具栏是 GitHub Release 的核心价值，用户已习惯；移除会降低体验 | 需要维护两套注入逻辑（通用 + GitHub 特殊），代码复杂度增加 |
| D7 | manifest.json 的 content_scripts.matches 使用 `["http://*/*", "https://*/*"]` | `<all_urls>` / 白名单 | 两种写法等价但前者更明确；`<all_urls>` 包含 file:// 等非 HTTP 协议（不需要）；白名单维护成本高 | 用户安装时会看到"读取所有网站数据"权限提示，需在商店说明 |
| D8 | 识别规则硬编码在代码中 | 配置文件 / 用户自定义 | v1 阶段硬编码最简单，满足 90% 场景；配置文件增加复杂度；用户自定义留给 v2 | 修改规则需要发布新版本，但 v1 规则已覆盖常见场景 |
| D9 | 不使用 HEAD 请求检测 MIME 类型 | 使用 HEAD 请求 | HEAD 请求增加网络开销（每个链接 1 次请求），可能被服务器拒绝或限流；组合策略已足够准确 | 某些仅通过 MIME 类型判断的下载链接会漏检，但可通过右键菜单兜底 |
| D10 | 错误处理采用静默失败 + 按钮状态反馈 | 弹窗提示 / 控制台日志 | 弹窗打扰用户；控制台日志普通用户看不到；按钮状态反馈直观且不打扰 | 开发者调试时需要手动添加 console.log |

## 3. 关键状态机

### 3.1 按钮状态机

```
状态：
- IDLE: "⚡ 加速下载" (enabled)
- LOADING: "⏳ ..." (disabled)
- SUCCESS: "✅ 已开始" (disabled)
- FAILED: "❌ 失败" (disabled)

转移：
IDLE ──[用户点击]──> LOADING
LOADING ──[API 成功]──> SUCCESS ──[2 秒后]──> IDLE
LOADING ──[API 失败]──> FAILED ──[2 秒后]──> IDLE
```

### 3.2 注入生命周期

```
状态：
- UNINITIALIZED: 页面未加载
- SCANNING: 扫描 DOM 中
- INJECTED: 已注入按钮
- OBSERVING: MutationObserver 监听中

转移：
UNINITIALIZED ──[DOMContentLoaded]──> SCANNING
SCANNING ──[扫描完成]──> INJECTED
INJECTED ──[启动 Observer]──> OBSERVING
OBSERVING ──[检测到新链接]──> SCANNING (增量)
```

## 5. 风险

| # | 风险 | 影响 | 概率 | 缓解 |
|---|---|---|---|---|
| R1 | 某些网站的 CSP 策略阻止注入按钮 | 按钮无法显示 | 低 | Content Script 默认不受 CSP 限制；如遇到，自动降级为仅右键菜单 |
| R2 | 识别规则误检，在非下载链接上注入按钮 | 用户体验下降，点击按钮无效 | 中 | 采用保守策略（宁可漏检不误检）；v1 上线后收集反馈，v2 优化规则 |
| R3 | 某些网站动态修改 DOM，导致已注入按钮被移除 | 按钮消失，用户需刷新页面 | 中 | MutationObserver 会重新检测并注入；如频繁发生，考虑 v2 增加重注入逻辑 |
| R4 | 布局冲突检测不完善，仍有部分网站布局错乱 | 页面显示异常 | 中 | 自动降级机制可覆盖 80% 情况；v1 上线后收集问题网站，v2 优化检测逻辑 |
| R5 | 用户对 `<all_urls>` 权限有隐私顾虑，拒绝安装 | 用户流失 | 高 | 在 Chrome Web Store 描述中明确说明：仅用于检测下载链接，不收集数据；提供隐私政策链接 |
| R6 | 性能问题：某些网站有 1000+ 链接，即使有上限仍可能卡顿 | 页面响应慢 | 低 | 链接数量上限 100 + 防抖 300ms 已覆盖大部分场景；如遇极端情况，考虑 v2 增加"禁用自动注入"选项 |
| R7 | 长期债务：硬编码识别规则难以维护 | 每次新增规则需发布新版本 | 高 | v1 接受此代价；v2 考虑引入配置文件或用户自定义规则 |

## 6. 不在范围

本次设计**不解决**但未来需要考虑的问题：

- **用户自定义识别规则**：v1 硬编码规则，v2 考虑允许用户添加文件扩展名或 URL 模式
- **排除网站列表**：v1 无法禁用特定网站的自动注入，v2 考虑增加黑名单功能
- **识别统计面板**：v1 不显示当前页面检测到的链接数，v2 考虑在 Popup 中增加统计功能
- **调试模式**：v1 无可视化调试工具，v2 考虑增加"高亮所有检测到的链接"功能
- **失败日志**：v1 加速失败时仅显示按钮状态，v2 考虑在 Popup 中记录失败日志
- **更精确的布局冲突检测**：v1 使用简单启发式，v2 考虑引入更复杂的布局计算
- **动态 URL 支持**：v1 不支持 `blob:` 或 `data:` URL，未来可考虑特殊处理（如提示用户）
- **多语言支持扩展**：v1 仅支持中文和英文，v2 考虑增加更多语言

## 9. 架构沉淀建议

### 9.1 新增的可复用抽象

| 路径 | 能力 | 触发场景 | 复用建议 |
|---|---|---|---|
| `src/content.js` 的 `LinkDetector` 模块 | 通用下载链接识别（组合策略） | 任何需要识别下载链接的场景 | 以后如需支持其他类型链接（如视频、图片），可扩展此模块 |
| `src/content.js` 的 `debounce()` 函数 | 防抖工具函数 | MutationObserver 回调 | 以后其他需要防抖的场景可复用（如 popup.js 的搜索输入） |
| `src/content.js` 的 `checkLayoutConflict()` 函数 | 布局冲突检测 | 注入任何 DOM 元素前 | 以后如需注入其他 UI 元素（如提示框），可复用此逻辑 |

### 9.2 新增 / 改变的项目级技术决策

| 决策 | 取值 | 影响范围 | 推翻代价 |
|---|---|---|---|
| Content Script 注入范围 | `<all_urls>`（所有 HTTP/HTTPS 网站） | 所有 content script 功能 | 高 — 用户已授权此权限，推翻需重新申请权限且可能流失用户 |
| 链接识别策略 | 组合策略（扩展名 OR URL 特征 OR download 属性） | 所有自动注入场景 | 中 — 修改策略需重新测试所有下载站，但不影响用户数据 |
| 性能上限 | 每页最多处理 100 个链接 | 所有自动注入场景 | 低 — 可通过配置调整，不影响核心逻辑 |

### 9.3 新增 / 修改的跨模块契约
