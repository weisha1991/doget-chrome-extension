## Why · 为什么做这个变更

### 问题陈述

当前 DoGet 扩展的自动注入功能**仅限于 GitHub Release 页面**，用户在其他下载站（如 SourceForge、GitLab Releases、npm registry、各类软件官网）访问时，只能通过以下方式加速：
1. 右键菜单（需要精确右键点击链接）
2. Popup 手动输入（需要复制 URL 再粘贴）

这两种方式都需要额外操作，不如 GitHub Release 页面的"一键加速"体验流畅。

### 业务价值

- **用户体验提升**：在任何下载站都能享受"看到即加速"的便捷体验
- **使用场景扩展**：从"GitHub Release 专用工具"升级为"通用下载加速器"
- **竞争力增强**：覆盖更广泛的下载场景，提升扩展的实用价值

### 目标用户

- 开发者：经常从 GitHub、GitLab、npm、PyPI 等平台下载工具和依赖
- 普通用户：从软件官网、开源镜像站下载安装包
- 所有需要频繁下载文件的用户

## What · 做什么

### 核心功能

**通用下载链接检测与加速注入**

在所有网站上智能检测下载链接，自动注入"⚡ 加速下载"按钮，提供与 GitHub Release 页面一致的加速体验。

### 功能详情

#### 1. 智能链接识别（组合策略）

使用多维度策略判断链接是否为"可加速的下载链接"：

**策略 A：文件扩展名匹配**
- 压缩包：`.zip`, `.tar.gz`, `.7z`, `.rar`, `.bz2`, `.xz`
- 可执行文件：`.exe`, `.dmg`, `.deb`, `.rpm`, `.apk`, `.msi`
- 安装包：`.pkg`, `.AppImage`, `.flatpak`, `.snap`
- 源码包：`.tar`, `.tgz`
- 其他：`.iso`, `.img`, `.bin`

**策略 B：URL 特征匹配**
- 路径包含：`/download/`, `/releases/`, `/dist/`, `/files/`, `/attachments/`
- 域名特征：`cdn.`, `dl.`, `download.`, `files.`, `releases.`
- 查询参数：`?download=`, `?file=`, `?attachment=`

**策略 C：HTML 属性检测**
- `<a download>` 属性存在
- `Content-Type` 响应头为 `application/octet-stream` 或其他下载类型（需要 HEAD 请求，按需使用）

**组合逻辑**：
- 命中策略 A **或** 策略 B → 标记为"高置信度下载链接"
- 仅命中策略 C → 标记为"中置信度下载链接"
- 都不命中 → 不注入按钮

#### 2. 通用注入机制

**注入形式**：在每个检测到的下载链接旁边注入独立按钮（复用 GitHub Release 的 UI 样式）

**触发范围**：智能检测，只在检测到下载链接时才注入（不预先声明网站白名单）

**性能优化**：
- 设置检测上限：每页最多处理 **100 个链接**
- 使用 **防抖/节流**：MutationObserver 触发后延迟 300ms 再批量处理
- **增量注入**：只处理新增的 DOM 节点，避免重复扫描
- **布局冲突检测**：注入前检查按钮位置是否会遮挡原有元素，冲突时自动降级为仅右键菜单

#### 3. 功能整合

**现有三个功能全部支持所有链接**：

| 功能 | 当前支持 | 变更后支持 |
|------|---------|-----------|
| 右键菜单 | 所有链接 | 所有链接（保持不变）|
| Popup 手动输入 | 所有 URL | 所有 URL（保持不变）|
| 自动注入按钮 | 仅 GitHub Release | **所有网站的下载链接** |

**架构调整**：
- 将 `src/content.js` 中的 GitHub 特定逻辑抽象为通用注入框架
- GitHub Release 页面作为"特殊场景"保留批量加速工具栏
- 其他网站使用通用注入逻辑（单个按钮 + 链接识别）

#### 4. 错误处理与用户反馈

**识别失败**：
- 用户可通过右键菜单或 Popup 手动触发加速（兜底方案）
- 不显示错误提示（避免打扰）

**加速失败**：
- 按钮状态变为 "❌ 失败"，2 秒后恢复
- 可选：在 Popup 中显示失败日志（供用户排查）

**布局冲突**：
- 自动降级为仅右键菜单，不注入可见按钮
- 在扩展设置中记录冲突网站（供用户查看）

## 影响面分析

### 代码影响

| 文件 | 变更类型 | 影响程度 |
|------|---------|---------|
| `src/content.js` | 重构 | 高（需要抽象通用注入框架）|
| `src/content.css` | 扩展 | 中（需要适配不同网站的布局）|
| `manifest.json` | 修改 | 高（需要调整 `content_scripts.matches` 为 `<all_urls>`）|
| `src/background.js` | 无变更 | 低（加速逻辑保持不变）|
| `src/popup.js` | 小幅扩展 | 低（可选：添加识别统计功能）|

### 权限影响

**需要新增权限**：
- `host_permissions: ["<all_urls>"]` 或 `["http://*/*", "https://*/*"]`

**用户感知**：
- Chrome Web Store 会提示"此扩展可以读取和更改您访问的所有网站上的数据"
- 需要在扩展描述中明确说明：仅用于检测下载链接，不收集用户数据

### 性能影响

- **内存**：每页增加 ~1-2MB（存储检测到的链接 Map）
- **CPU**：MutationObserver + 链接扫描，预计增加 5-10% CPU 占用
- **网络**：无额外请求（不使用 HEAD 请求检测 MIME 类型）

### 用户体验影响

**正面**：
- 在所有下载站都能享受一键加速
- 减少右键菜单和 Popup 的使用频率

**潜在风险**：
- 某些网站可能因注入按钮导致布局错乱（已有降级方案）
- 权限提示可能让部分用户担心隐私（需要在商店页面说明）

## 范围排除 · 本次不做什么

### 明确不做

1. **不支持需要登录才能下载的资源**
   - 理由：需要处理 Cookie、Session，复杂度高且涉及隐私
   - 兜底：用户可通过 Popup 手动输入 URL

2. **不处理动态生成的下载链接**
   - 如：点击按钮后 JS 生成的 `blob:` URL 或 `data:` URL
   - 理由：这类链接通常是临时的，DoGet API 无法加速
   - 兜底：用户可通过浏览器原生下载

3. **不做下载管理器功能**
   - 如：断点续传、多线程下载、下载队列
   - 理由：超出"加速代理"的核心定位，且 Chrome 已有原生下载管理器
   - 未来：可考虑与第三方下载管理器集成

4. **不做浏览器兼容**
   - 仅支持 Chrome/Edge（Manifest V3）
   - 不支持 Firefox（需要 Manifest V2 或 V3 的 Firefox 版本）
   - 理由：Firefox 的 Manifest V3 支持尚不完善

### 边界说明

- **识别策略**：优先准确率，宁可漏检也不误检（避免在非下载链接上注入按钮）
- **性能上限**：每页最多处理 100 个链接，超出部分不注入（但右键菜单仍可用）
- **布局冲突**：检测到冲突时自动降级，不强制注入
